# 论文插图落地说明

last_updated: 2026-04-11

## 适用范围

本说明服务于论文 [`../Money_Is_All_You_Need_final_CN.md`](../Money_Is_All_You_Need_final_CN.md) 的全部插图重绘与后续维护。

当前目标不是重写论文，而是把图表统一收敛为“客观事实 + 清晰结构 + 极少解释文字”的表达方式。结论写在正文里，不写在图里。

## 全局绘制规则

- 所有图统一移除左上角“图 1 / 图 2 / Figure 1”之类的额外标签。
- 图内只保留标题、客观数据、结构关系、关键角色、关键路径。
- 不在图内写长段落结论，不额外加总结区。
- 文案尽量压到一句标题 + 少量节点标签。
- 优先使用图标、节点、直角折线、虚线边界和清晰分层，不使用弯曲混乱连线。
- 线条尽量横平竖直；需要跳跃关系时用虚线，不用随意斜拉。
- 图内所有数字、标题和标签必须避免溢出、遮挡和压线。
- 每张图优先展示“事实结构”，不展示“作者结论”。
- 所有图最终文件统一保存在当前目录，并由论文正文直接引用 `.svg`。

## fig01 开发者人口比例

### 这张图要表达什么

OpenClaw 的热度主要发生在极少数高强度参与者与开发者群体内部，不等于复杂 Agent 已经大众化。

### 图上必须出现的元素

- 三层同心圆
- `81 亿`：最外层，代表全球人口
- `1.8 亿`：中间层，代表全球开发者
- `35.1 万`：最内层，代表 OpenClaw 的高强度公开参与信号

### 具体改造要求

- 去掉图内结论文案
- 让三个数字都放在对应圆层内，不能压线、不能溢出
- 外圈最醒目但不喧宾夺主，中圈蓝色，内圈更深色
- 标题只描述客观事实，不替正文下判断

## fig02 AI Agent 与人类社会特征类比

### 这张图要表达什么

AI Agent 已经开始具备一部分人类社会主体的基础能力，因此“像主体一样进入社会分工”具备讨论基础。

### 图上必须出现的元素

- 左侧：人类世界
- 右侧：AI Agent 世界
- 四组左右映射
- 人类侧标签至少包含：
  - 短期记忆
  - 长期记忆
  - 使用软件 / APP
  - 协作分工
- Agent 侧对应标签至少包含：
  - memory
  - persistent memory / profile
  - skills / tools / API
  - multi-agent / runtime coordination

### 具体改造要求

- 标题改为“AI Agent 已经具备了一些人类社会的特征”
- 删除“真正的差别”之类结论句
- 删除过多小字
- 整体变成一张左右映射关系图，不是论文摘要图

## fig03 人类社会与 AI 社会通路断点

### 这张图要表达什么

人类社会中“买软件”和“买服务”两条通路都成熟；AI 世界里“本地 Agent 向外购买结果服务”这条路仍然缺失。

### 图上必须出现的元素

- 左侧：人类社会
- 右侧：AI 世界
- 人类世界两条路：
  - 买软件 -> 结果
  - 买服务 -> 其他人使用软件 -> 结果
- 结果回到人类
- AI 世界路径：
  - 主人 -> 本地 AI Agent（执行主体）
  - 安装 runtime
  - AI Agent -> 找 skill / API -> 结果
  - 本地 AI Agent -> 购买别的 Agent 服务，这条线断裂或缺失

### 具体改造要求

- 人类侧和 AI 侧必须并排对照
- 线条要直、关系要清楚
- 不保留一堆解释小字
- 断裂通路要明确看得见

## fig03b Money 作为最小市场协议

### 这张图要表达什么

Money 在本文里不是“付钱一下”，而是一套让 Agent 真正能够进入市场交易的最小协议。

### 图上必须出现的元素

- 一个顺序式结构图
- 六个步骤或六个节点：
  - 预算授权
  - 价格发现
  - 履约边界
  - 验收证据

### 具体改造要求

- 标题直接写“Money 是一套市场协议”
- 六个点之间有顺序连接
- 去掉长解释
- 只保留最小必要词组

## fig04 现有生态的两条路线

当前生态主要只有两条路线：自己安装 skill，或者直接采用整包 AI Agent / 云端高手。

### 图上必须出现的元素

- 路线 A：自己练功 / 装 skill
  - 主人
  - AI Agent
  - skill 市场
  - 安装 / 配置
  - 执行结果
- 路线 B：整包高手 / 整包龙虾
  - 主人
  - 整包 Agent
  - 内置 skills / workflows
  - 执行结果

### 具体改造要求

- 两条路线并列
- 玩家与流程都要看得清
- 不重复写正文结论

## fig05 多入口专用 Agent 与单一大管家

### 这张图要表达什么

多个专用入口会让主人重复喂信息、重复同步上下文；更合理的是一个大管家统一调度。

### 图上必须出现的元素

- 左侧：主人分别对接多个专用 Agent
- 右侧：主人只对接一个大管家，大管家再调度多个执行者
- 信息、预算、品牌手册、审批规则等上下文在左侧重复分发，在右侧一次输入

### 具体改造要求

- 线条清楚
- 左右对照鲜明
- 文字减少，用图标表达“重复输入”与“统一调度”

## fig06 OpenSlaw 的平台位置

OpenSlaw 不是 skill 库，而是夹在需求侧大管家与供给侧市场之间的结果交易平台。

### 图上必须出现的元素

- 主人
- 需求侧大管家 Agent
- OpenSlaw 平台
- 供给方市场 / 供给方 Agent
- 订单、交付、证据、评价、结算这几类平台能力

### 具体改造要求

- 平台与社区要区分，不要混为一层
- 平台承接交易事实，不承接供给方实际执行

### 这张图要表达什么

当没有现成商品时，大管家发布需求，由多个供给方 Agent 查看、报价、接单，再由需求侧确认。

### 图上必须出现的元素

- 大管家发布需求
- 平台需求池 / brief
- 多个供给方 Agent 查看需求并回提案
- 需求侧确认需求包与提案

- 强调需求确认在需求侧，大管家不是直接把半成品扔出去

### 这张图要表达什么

### 这张图要表达什么

短期内同一个需求会同时存在两条通路：人类找真人完成，或 AI Agent 找 AI-native 路径完成。

### 图上必须出现的元素

- 共同起点：视频剪辑需求
- 路线 A：找真人 -> 使用某个视频软件 -> 结果
- 路线 B：找 AI Agent -> 仍然使用同类软件或能力栈 -> 结果

## fig11 中期：搜索、比较、选供给

### 这张图要表达什么

中期能力不是只找一个“视频剪辑”通用入口，而是 AI Agent 能根据更细分需求去搜索、比较并选中最合适的供应商。

### 图上必须出现的元素

- 左上有人类对照：
  - 人类需求 -> 找一个泛化“视频剪辑”
- 右侧 AI 路径：
  - 主人需求 -> AI Agent
  - AI Agent 先细化需求
  - 细化后的需求分别连向三个不同风格 / 专长的供应商
  - 其中一个被高亮选中

### 具体改造要求

- 强化“更细分供给”的概念
- AI Agent 要先返身理解需求，再向市场搜索
- 图里不要把市场画成自动装配器

## fig12 长期：需求拆解与编排

### 这张图要表达什么

长期真正稀缺的是需求侧 AI Agent 的拆解与编排 intelligence，它会把一个复杂需求拆成多个可购买结果，再在自己这一侧组装。

### 图上必须出现的元素

- 主人 -> AI Agent
- AI Agent 拆出多个子任务
- 每个子任务单独连到平台上的对应供给
- 多个结果全部回到 AI Agent
- AI Agent 完成最终装配后交给主人

### 具体改造要求

- 删掉“主人直接找人”的旧路径
- 装配动作必须发生在 AI Agent 一侧，不在市场里
- 每条拆解线都要清楚

## fig13b 人类评价与 AI Agent 公允评价

### 这张图要表达什么

人类评价依赖主观总结，因此“评价动作”本身必要；AI Agent 世界可以直接读取交易快照，因此未必依赖同样的主观评价结构。

### 图上必须出现的元素

- 左侧：人类评价体系
  - 标注“评价必要”
  - 原因关键词：无法读完整交付历史、只能看总结、信息不完整
- 右侧：AI Agent 评价体系
  - 标注“评价不再是唯一必要入口”
  - 可读取的快照内容：
    - 需求摘要
    - 正式输入
    - 交付摘要
    - 正式输出
    - 评价快照

### 具体改造要求

- 删除结论段
- 两侧对照必须清楚
- 快照项保留，其他赘述删掉
