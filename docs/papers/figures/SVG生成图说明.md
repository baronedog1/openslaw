## 适用范围

本说明服务于论文 [`../Money_Is_All_You_Need_final_CN_watermarked.pdf`](../Money_Is_All_You_Need_final_CN_watermarked.pdf) 的全部插图生成、微调、定版与后续维护。

本文档现在不是单纯的“插图落地记录”，而是三件事的统一事实来源：
- `figures/` 目录当前定版 SVG 的事实说明
- `figure-canvas-editor.html` 的编辑规范
- `scripts/generate_paper_figures.mjs` 以后必须遵守的生成规范

## 当前事实来源

- 当前目录里的 `fig01` 到 `fig15` 共 `21` 张 `fig*.svg`，现在都已经是“经过 HTML 微调后的定版文件”。
- 这些 `fig*.svg` 才是当前论文插图的正式事实来源，不再把“脚本刚生成出来的版本”视为默认终版。
- 如果将来脚本输出和当前目录里的定版 SVG 不一致，处理顺序必须是：
  1. 先以当前目录里的定版 SVG 为准
  2. 再回改 `scripts/generate_paper_figures.mjs`
  3. 让脚本重新向定版结构对齐
- 不允许长期维持“脚本版一套、定版 SVG 一套、HTML 里又一套”的三套口径。

## 目录说明

- `fig*.svg`
  - 论文正式插图文件
  - 也是 HTML 编辑器导入时的直接素材源
- `figure-canvas-editor.html`
  - 单文件本地编辑器
  - 用来拖动组件、编辑文字、重连线条、导出 `SVG / PNG / JSON`
- `figure-canvas-editor.assets.js`
  - 编辑器的内置资源包
  - 由生成脚本自动同步
- `components/`
  - 可复用组件库
  - 用于把常见图形拆成稳定的可编辑单元
- `scenes/`
  - 少数已经沉淀成场景模板的图
- `serve_figure_canvas_editor.*`
  - 可选辅助启动脚本
  - 不是唯一入口，也不是正式依赖
- `../../../scripts/generate_paper_figures.mjs`
  - SVG 与编辑器资源包的统一生成脚本

## 当前定版 SVG 相比早期生成稿的共同优化

以下几点，是这次你在 HTML 里人工微调后，已经明确体现出来、并且后续必须保留的方向：

- 排版更稳定
  - 标题、主图、说明区和底部留白关系更清楚
  - 不再接受“能表达但很挤”的草稿排版
- 组件边界更清楚
  - 大面板、卡片、标题块、角色图标都有稳定边界
  - 组件之间的层级关系更明确
- 线条逻辑更直
  - 连接线尽量横平竖直
  - 箭头从组件四边中点出入
  - 箭头头部更小，不压文字
- 文字可读性更好
  - 不再允许压线、重叠、溢出
  - 多行文本的换行、字号和行距更稳定
- 编辑粒度更细
  - 不只是整张图可拖动
  - 组件内部的文字、图标、外框也需要可单独编辑
- 导出边界更明确
  - 每张图都应该有最外层范围框
  - 导出时只导出这个框内的内容，不导出整页空白

## 全局绘制规则

- 所有图统一移除左上角“图 1 / 图 2 / Figure 1”之类的额外标签。
- 图内只保留标题、客观数据、结构关系、关键角色、关键路径。
- 不在图内写长段落结论，不额外加总结区。
- 文案尽量压到一句标题 + 少量节点标签。
- 优先使用图标、节点、直角折线、虚线边界和清晰分层，不使用弯曲混乱连线。
- 线条尽量横平竖直；需要跳跃关系时用虚线，不用随意斜拉。
- 所有连线只能从组件四个边的中点出发或落入，不允许从任意位置穿出组件。
- 一个组件要连向哪个方向，就必须从朝向该方向的边中点出线；进入组件时也必须落到对应方向的边中点。
- 箭头头部要小，不得压住卡片、图标或文字本体。
- 图内所有数字、标题和标签必须避免溢出、遮挡和压线。
- 每张图优先展示“事实结构”，不展示“作者结论”。
- 所有图最终文件统一保存在当前目录，并由论文正文直接引用 `.svg`。

## 以后所有 SVG 的生成规则

以后不允许再先生成“只适合展示、不适合编辑”的 SVG。新的论文图必须从一开始就按“可进 HTML 编辑器继续改”的结构生成。

### 必须满足的结构要求

- 每张图都必须带独立 `export-frame`
  - 这是导出边界
  - 也是画布里的最外层范围框
- 每张图都必须带独立 `title-block`
  - 标题和标题下横线不能散落成零碎元素
- 每个逻辑组件都必须是顶层 `<g>`
  - 使用 `data-editor-group="component"`
  - 使用 `data-editor-name="..."` 标明组件名
- 组件内部仍需继续可拆
  - 使用 `data-editor-subitem="true"`
  - 至少拆出：外框、图标、文字块
- 所有文字必须保留为 `<text> / <tspan>`
  - 不能转路径
  - 不能只靠外部 CSS 伪元素显示
  - 每段文字都应带明确 `x / y`
- 正式连线必须是顶层 `line / polyline / path`
  - 有箭头时使用 `marker-end`
  - 不能藏在难以恢复的复杂蒙版、滤镜、外链样式里
- 无箭头但有结构意义的线也必须保留为独立可见元素
  - 不能在导入可编辑版时消失
- 样式应尽量写成元素内联属性
  - 如 `fill / stroke / stroke-width / opacity / text-anchor`
  - 不要把关键显示逻辑依赖在外部 CSS 文件上
- 顶层元素顺序就是默认层级顺序
  - 不要依赖导入后再猜层级

### 生成主线

以后每张图都按下面流程走：

1. 先由 `scripts/generate_paper_figures.mjs` 生成可编辑结构的 SVG
2. 再进 `figure-canvas-editor.html` 做人工微调
3. 微调后的结果直接覆盖当前目录里的 `fig*.svg`
4. 再把这些定版差异回写进生成脚本与组件 helper

也就是说：
- `HTML 微调` 是正式流程的一部分
- 但 `HTML 微调结果` 不能只停留在手工文件里
- 最终必须回流到生成规则里

## figure-canvas-editor.html 的职责

### 入口文件

- 编辑器 HTML：`figure-canvas-editor.html`
- 内置资源包：`figure-canvas-editor.assets.js`
- 生成脚本：`../../../scripts/generate_paper_figures.mjs`
- 可选本地启动脚本（Windows）：`start-figure-canvas-editor.bat`
- 可选本地启动脚本（Node）：`serve_figure_canvas_editor.mjs`
- 可选本地启动脚本（Python）：`serve_figure_canvas_editor.py`

### 当前应支持的编辑能力

- 把当前目录内的论文插图 SVG 作为“可拖动组件”加载到画布
- 在画布中移动、复制、缩放这些 SVG 组件
- 以“四边中点”为唯一端口重新连接线条
- 直接在组件里双击文字编辑
- 直接拖动文字在组件内的位置
- 直接拖动组件内部图标子组
- `Ctrl/Cmd + C` 与 `Ctrl/Cmd + V` 复制粘贴选中的组件与它们之间的连线
- 支持创建空白文字组件，作为新的卡片/标注起点
- 导出当前排版结果为 `JSON`、新的整张 `SVG` 或 `PNG`

### 使用方式

1. 先运行 `node scripts/generate_paper_figures.mjs`
2. 打开 `figure-canvas-editor.html`
3. 从左侧素材栏把内置 SVG 拖进画布，或导入本地 `.svg / .json`
4. 选中组件后可移动、复制、缩放
5. 在画布空白处拖动鼠标可框选多个组件；按住 `Shift` 点击组件可增减多选
6. 每张图最外层的 `export-frame` 是范围框；导出以它为裁切边界
7. 直接连续点击两个组件的四边中点即可创建箭头连线
8. 右键组件可调整层级；右键连线可改颜色、粗细、虚线 / 实线
9. 双击组件里的文字，可直接在画布上改字；拖动该文字，可直接改它在组件里的位置
10. 顶部可直接新建空白文字组件，用作标题、备注或补充模块
11. 完成后可一键导出 `JSON`、`SVG` 或 `PNG`

## 当前统一规范

- 所有顶层 `fig*.svg` 必须走同一套“自动拆组 -> 组件级编辑 -> 裁切导出”链路。
- 每张图进入画布后，默认应尽量保持原图排版关系；不允许插入后就自动散架。
- 每个组件至少要支持：
  - 拖动位置
  - 等比缩放
  - 层级调整
  - `Ctrl/Cmd + C / Ctrl/Cmd + V` 复制粘贴
  - 文字编辑
  - 组件内文字直接双击编辑
  - 组件内文字位置拖动
  - 组件内图标子组拖动
- 每条线至少要支持：
  - 颜色
  - 粗细
  - 实线 / 虚线
  - 重新连接起点 / 终点
- 导出 `SVG / PNG` 时只导出 `export-frame` 范围，不导出整页空白画布。

## 当前定版 SVG 已经体现出来的技术特征

对当前目录里 21 张正式图做结构扫描后，已经可以确认这些共同点：

- 每张图都已经带 `export-frame`
- 每张图都已经带 `title-block`
- 每张图都已经保留大量 `<text>` / `<tspan>`
- 大多数图都已经把核心组件拆成 `data-editor-group="component"`
- 大多数组件内部也已经拆成 `data-editor-subitem`
- 带箭头的图已经大量使用 `marker-end`

这意味着后续脚本生成时，不能再退回到“整张大 SVG 只有路径、没有组件语义”的旧方式。

## 当前约束与处理原则

- 当前所有顶层 `fig*.svg` 都按同一套自动拆组逻辑进入画布，不再对个别图长期保留特殊模板优先。
- 如果某张图导入后失真，优先修图本身的分组与结构，不优先给编辑器加特判。
- 对于带箭头但当前无法稳定还原成边的线，必须退回成独立线组件，先保证原样可见，再允许后续手工重连。
- 只有最外层 `export-frame` 节点按“画布底板”处理；标题、panel、card 和普通说明组本身仍然应该可选中、可编辑。
- 单个组件里已有的文本必须支持两种编辑方式：右侧属性面板直接编辑，或在组件里双击文字直接编辑。

## fig03 人类社会与 AI 社会通路断点

### 这张图要表达什么

人类社会中“买软件”和“买服务”两条通路都成熟；AI 世界里“本地 Agent 向外购买结果服务”这条路仍然缺失。

### 图上必须出现的元素

- 左侧：人类社会
- 右侧：AI 世界
- 人类世界两条路：
  - 买软件 -> 结果
  - 买服务 -> 其他人使用软件 -> 结果
- 结果回到人类
- AI 世界路径：
  - 主人 -> 本地 AI Agent（执行主体）
  - 安装 runtime
  - AI Agent -> 找 skill / API -> 结果
  - 本地 AI Agent -> 购买别的 Agent 服务，这条线断裂或缺失

### 具体改造要求

- 人类侧和 AI 侧必须并排对照
- 人类侧固定为两条成熟通路，并且都回到同一个“结果”节点：
  - 买软件 -> 结果 -> 人类
  - 买服务 -> 其他人使用软件 -> 结果 -> 人类
- 结果节点应放在中部偏下位置，形成两条路汇聚
- AI 侧要基于人类图做结构映射：
  - 右侧节点数量、相对位置和左侧保持同构，先复用左图骨架，再在此基础上替换含义
  - 人类执行主体 -> 本地 AI Agent
  - 人类本人 -> 主人，并且主人位置上移
  - `runtime` 作为进入 AI Agent 的前置条件，放在左上位，并且连线必须指向 Agent，而不是从 Agent 发出
  - `找 skill / API` 放到 `结果` 正上方，这样上方成熟通路是 `Agent -> skill / API -> 结果`
  - 下方保留与左图同位置的“买别的 Agent 服务”路径，但直接在这个组件上打叉，不再继续画出完整已成立通路
- 线条要直、关系要清楚，优先只保留两条主干，不做多余绕线
- 上下两条路径都要留出明确的水平走廊，任何箭头都不能压到卡片文字
- 不保留多余解释小字
- 断裂通路要明确看得见

多个专用入口会让主人重复喂信息、重复同步上下文；更合理的是一个大管家统一调度。

### 图上必须出现的元素

- 左侧：主人分别对接多个专用 Agent
- 右侧：主人只对接一个大管家，大管家再调度多个执行者
- 信息、预算、品牌手册、审批规则等上下文在左侧重复分发，在右侧一次输入

### 具体改造要求

- 线条清楚
- 左右对照鲜明
- 文字减少，用图标表达“重复输入”与“统一调度”
